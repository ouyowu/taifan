# Hermes Supervision SOP

适用项目：ThaiStar Bridge / 泰娱桥  
项目路径：`/Users/ouyowu/Documents/泰国娱乐圈项目`

## 1. 目标

这份 SOP 用于指导 Hermes agent 如何监督 Codex 在本项目上进行“小步、可验证、低风险”的持续迭代。

核心原则：

- 只给 Codex 小任务
- 每次只动一个明确范围
- 每次都要有可见验收标准
- 每次完成后都要人工复核
- 不允许 Codex在单轮里做大范围重构

## 2. 为什么要这样监督

这个项目目前已经不是空白壳子，而是：

- 前台多页面已可运行
- 后台运营流已较复杂
- AI 抓取、摘要、提取、草稿回填都已打通
- 批处理、推荐方向、自动拆批、上下文保留都已经存在

风险点在于：

- 单文件过大，尤其是 `src/components/admin-console.tsx`
- 首页提示内容累积较多
- 语言展示规则必须保持一致
- AI / 批处理流一旦大改，很容易出现边界破坏

所以 Hermes 的职责不是“直接让 Codex 做大重构”，而是把工作拆成一串足够小、每步都能确认没坏的任务。

## 3. Hermes 的工作方式

每次只做下面 5 步：

1. 选一个最小可验证任务
2. 给 Codex 明确边界
3. 让 Codex 返回结果
4. 人工验收
5. 再决定下一步

不要跳步。  
不要一次给 Codex 一个“顺手把其他相关部分也一起优化”的任务。

## 4. 给 Codex 任务的统一模板

每次任务都应该至少包含这 6 部分：

1. 工作目录
2. 本轮只允许动的文件或区域
3. 必须参考的上下文文件
4. 必须保留的现有能力
5. 输出要求
6. 验证要求

推荐模板：

```md
请基于 `/Users/ouyowu/Documents/泰国娱乐圈项目` 完成一个小范围任务。

任务目标：
<只写一个目标>

必须先参考：
- <文件 1>
- <文件 2>
- <文件 3>

限制：
- 只改 <指定文件/区域>
- 不要顺手改别的模块
- 不要扩大范围
- 必须保留 <现有能力 A/B/C>

输出要求：
1. 直接修改代码
2. 说明改了哪些文件
3. 说明为什么这样做
4. 给出验证结果

验证要求：
- `npm run lint`
- `npx next build --webpack`
```

## 6. 当前推荐监督顺序

这是目前最稳的顺序：

1. `admin-console.tsx` 结构审查与拆分提案
2. 选择一个最适合先拆的模块做最小重构设计
3. 先拆抓取任务队列
4. 再拆活动表单
5. 再拆动态表单
6. 再拆明星优先级列表
7. 收敛首页过量提示卡
8. 评估是否升级到双字段模型

注意：

- 第 1、2、8 步可以只做分析，不动代码
- 第 3 到 7 步才适合真实改代码

## 7. 人工验收总清单

每次 Codex 完成后，Hermes 先看这 8 件事：

1. 是否只改了指定范围
2. 是否顺手动了无关文件
3. `npm run lint` 是否通过
4. `npx next build --webpack` 是否通过
5. 现有功能有没有少
6. 现有文案语义有没有被改偏
7. “中文解释 + 原文保留”规则有没有被破坏
8. 复杂状态流有没有被拆乱

## 8. 分模块验收重点

### 8.1 抓取任务队列

必须保留：

- 来源筛选
- 状态筛选
- 单条推荐方向
- 推荐按钮高亮
- 批量推荐处理
- 混合批次提醒
- 自动拆成两批
- 活动批结束自动切动态批
- 批处理顶部上下文提示
- 会话级保留筛选和勾选

### 8.2 活动表单

必须保留：

- 新增
- 编辑
- 保存并发布
- 关联明星
- 活动草稿批处理
- 草稿上下文提示
- 示例写法
- 原文标题 + 中文摘要规则

### 8.3 动态表单

必须保留：

- 新增
- 编辑
- 保存并发布
- AI 动态草稿回填
- 批处理上下文提示
- 原始标题 / 官方写法
- 中文摘要
- 中文整理
- 原文保留建议

### 8.4 明星优先级列表

必须保留：

- `TOP` 排序显示
- 上移 / 下移
- 设为 `TOP 1`
- 重置推荐排序
- 机构筛选
- 标签筛选
- 只看高优先级
- 导出 / 复制按钮
- 中文称呼 + 官方原文名逻辑

### 8.5 首页收敛

重点不是“加更多内容”，而是：

- 减少重复提示
- 保留关键入口
- 提高清晰度
- 保住新粉友好定位

## 9. Hermes 最该拦下来的情况

如果 Codex 出现下面任一情况，Hermes 应直接打回：

1. 一次改太多模块
2. 顺手做了未要求的重构
3. 没跑验证
4. 语言展示规则被弱化
5. 批处理 / AI 流程被改得更难理解
6. 首页继续膨胀而不是收敛
7. 把父组件关键状态拆散，导致行为更难追踪

## 10. Hermes 给 Codex 的任务粒度标准

合格任务通常符合：

- 预计 < 5 分钟可完成
- 只动 1 个模块
- 修改文件数量少
- 可以立刻验证
- 人工能快速判断是否成功

不合格任务通常长这样：

- “顺手优化一下后台结构”
- “把首页和后台一起整理”
- “统一修一下所有语言展示”
- “把所有表单都拆掉”

这些任务太大，Hermes 不应直接交给 Codex。

## 11. Hermes 对 Codex 输出的要求

Hermes 应坚持要求 Codex 最后必须说明：

- 改了哪些文件
- 为什么这样改
- 哪些现有能力被保留
- 如何验证
- `lint` / `build` 结果

如果 Codex 没写这些，Hermes 要追问，而不是直接继续下一轮。

## 12. 当前项目最容易继续出问题的地方

### 高风险

- `src/components/admin-console.tsx`
- 首页 `src/app/page.tsx`
- AI 草稿回填逻辑
- 批处理状态流

### 中风险

- 语言规则在列表页、详情页、后台录入页的落地一致性
- 单字段模型随着内容量增大后的维护性

## 13. 给 Hermes 的最终建议

Hermes 最适合扮演的角色不是“替 Codex 写代码”，而是：

- 严格缩小任务
- 盯住验收
- 阻止范围蔓延
- 保住项目已经建立好的产品方向

一句话原则：

> 先保住功能和规则，再做结构优化；先做小步验证，再做下一步。
